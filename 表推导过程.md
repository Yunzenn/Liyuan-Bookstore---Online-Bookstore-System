需求 → 表结构推理过程
第一步：从需求中提取名词（候选实体）
原始需求关键语句：

"新客户第一次登录书店Web站点前，需申请账号并登记注册信息，如客户名、客户地址和电话等，站点会给每个客户分配唯一标识号"

"客户可以在Web站点上浏览图书目录并对需要的书下订单。图书目录包括ISBN号、书名、作者、定价、出版年等"

"一个客户可以下多个订单，也可以在一个订单中订购多种不同的书，比如同时购买数据库教材30本和操作系统教材28本"

提取的名词	是否为实体	理由
客户	✅ 是	有独立属性（客户名、地址、电话），有唯一标识
图书/图书目录	✅ 是	有独立属性（ISBN、书名、作者、定价、出版年）
订单	✅ 是	有独立生命周期（状态流转），需跟踪管理
订单明细	✅ 是	"一个订单中订购多种不同的书"→ 需要记录每种书的数量
第二步：提取属性
客户实体属性推导：


需求原文："登记注册信息，如客户名、客户地址和电话等"
                ↓
        直接提取 → customer_name, address, phone

需求原文："站点会给每个客户分配唯一标识号"
                ↓
        需要主键 → customer_id

需求原文："新客户第一次登录"
                ↓
        需要登录凭据 → email（登录名）, password_hash（密码）

实际需要 → created_at（注册时间）, updated_at（更新时间）
图书实体属性推导：


需求原文："图书目录包括ISBN号、书名、作者、定价、出版年等"
                ↓
        直接提取 → ISBN, title, author, price, publish_year

实际需要 → publisher（出版社）, stock（库存数量，用于下单时校验）
订单实体属性推导：


需求原文："一个客户可以下多个订单"
                ↓
        需要关联客户 → customer_id（外键）

需求原文："订单、出货、货款交付状态"
                ↓
        需要状态字段 → status（枚举：待付款/已付款/已发货/已完成/已取消）

实际需要 → order_id（主键）, order_date（下单时间）, total_amount（订单总额）
订单明细实体属性推导：


需求原文："在一个订单中订购多种不同的书，比如同时购买数据库教材30本和操作系统教材28本"
                ↓
        "多种不同的书" → 需要ISBN
        "30本"、"28本" → 需要quantity（数量）
        
实际需要 → unit_price（下单时单价，因为价格可能变动）

主键推导 → 一个订单中的每种书唯一确定 → (order_id, ISBN) 联合主键
第三步：识别关系
关系推导过程：


需求原文："一个客户可以下多个订单"
                ↓
        客户(1) ──订单(N)  →  1:N关系
                ↓
        实现方式：Order表添加customer_id外键

需求原文："在一个订单中订购多种不同的书"
                ↓
        订单(N) ──图书(M)  →  M:N关系
                ↓
        实现方式：新建OrderItem关联表

需求原文："比如同时购买数据库教材30本和操作系统教材28本"
                ↓
        一个订单包含多条明细 → Order(1) ──OrderItem(N)
        一种书可出现在多个订单 → Book(1) ──OrderItem(N)
最终关系图：


Customer (1) ──────── (N) Order
                          │
                          │ (1)
                          │
                          ▼
                       OrderItem (N) ──────── (1) Book
第四步：推导外键和约束
外键推导：

外键字段	所在表	推导逻辑
customer_id	Order表	"一个客户下多个订单" → 订单需知道是谁下的
order_id	OrderItem表	"一个订单中多种书" → 明细需知道属于哪个订单
ISBN	OrderItem表	"订购不同的书" → 明细需知道是哪种书
约束推导：


需求："客户名、客户地址和电话"
        ↓
customer_name NOT NULL（注册必填）

需求："站点会给每个客户分配唯一标识号"
        ↓
customer_id PRIMARY KEY AUTO_INCREMENT

需求："第一次登录"（隐含需要账号）
        ↓
email UNIQUE NOT NULL（作为登录标识）

需求："定价"
        ↓
price DECIMAL(10,2)（金额用定点数，避免浮点误差）

需求："30本"
        ↓
quantity INT CHECK(quantity > 0)（数量必须为正整数）

需求："出货、货款交付状态"
        ↓
status ENUM('待付款','已付款','已发货','已完成','已取消')
第五步：验证设计是否满足需求
需求场景	验证SQL	是否满足
新客户注册	INSERT INTO customer (...) VALUES (...)	✅
系统分配唯一标识	customer_id AUTO_INCREMENT 自动分配	✅
浏览图书目录	SELECT * FROM book	✅
一个客户下多个订单	Order表通过customer_id关联	✅
一个订单多种书	OrderItem表通过(order_id, ISBN)关联	✅
购买数据库教材30本+操作系统教材28本	INSERT INTO order_item VALUES (order_id, 'ISBN1', 30, ...), (order_id, 'ISBN2', 28, ...)	✅
跟踪订单状态	Order表status字段 + 状态流转	✅
出货状态管理	UPDATE order SET status='已发货' WHERE order_id=?	✅
货款状态管理	UPDATE order SET status='已付款' WHERE order_id=?	✅
第六步：范式验证
为什么要满足3NF？


假设不满足3NF，存在传递依赖：
Order表：order_id → customer_id → customer_name

问题：如果客户改名，需要更新所有该客户的订单记录
      → 数据冗余 → 更新异常

解决：Order表只存customer_id，不存customer_name
      → 需要客户名时通过JOIN查询
      → 满足3NF ✅
unit_price为什么放在OrderItem而不是Book表？


需求场景：图书可能涨价或打折

如果放在Book表：
  - 订单记录的是当前价格，不是下单时价格
  - 价格上涨后，历史订单金额会变
  → 数据不一致

放在OrderItem表：
  - 记录下单时的价格快照
  - 历史订单金额不受价格变动影响
  → 满足业务需求 ✅
总结：推理链

需求文本
    ↓ 提取名词
识别实体（Customer, Book, Order, OrderItem）
    ↓ 提取属性词
确定各实体属性
    ↓ 分析"一个...多个..."句式
识别关系（1:N, M:N）
    ↓ M:N关系需拆解
设计关联表（OrderItem）
    ↓ 分析业务约束
确定外键、主键、CHECK约束
    ↓ 验证范式
确保3NF/BCNF，消除冗余
    ↓ 验证场景
用SQL模拟每个需求场景，确认可行
